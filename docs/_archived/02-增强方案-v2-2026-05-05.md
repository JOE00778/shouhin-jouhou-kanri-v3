## 模块 ① · 自动产品等级修改（季度 · Boss-only）

### 4 档定义（Boss 已确认）

| 档 | 判定规则 | 订货策略 | 估算 SKU 数 |
|---|---|---|---|
| **A** | 销售前 80% × 利润率 ≥ 59% | 主动备货，**不能断货**，按进货周期前下单 | ~200 |
| **B** | 销售前 80% × 利润率 < 59% | 按需补货，跟踪毛利改善空间 | ~1,000 |
| **C** | 销售后 20%（含 NEW 新品过渡）| 最低安全库存 / 缺货才补 / 不主动备 | ~5,000 |
| **停售** | NetSuite 取扱区分 ∈ ('取扱中止','メーカー取扱中止') OR 模块 ③ 改廃確認=取扱中止 | **完全不订货**，专注清库存 | ~1,500 |

### 联动订货公式（模块 #5 自动订货）

```python
# 安全系数按档差异化
SAFETY = {'A': 1.5, 'B': 1.0, 'C': 0.5, '停售': 0}

# 再订货点（库存低于此值就补货）
reorder_point = monthly_avg_sales * (lead_time_days / 30) * SAFETY[rank]

# 建议下单量
suggested_order_qty = monthly_avg_sales * min(lead_time_months, 3) * SAFETY[rank]
```

## 模块 ② · 一元化指标增强（健康度独立模型）

### a) 月/SKU 库存存销比（自动 · 月度）

```python
存销比_months = 月末在庫量 / 月销量
```

输出表 `stock_sales_ratio_monthly`：sku / year_month / end_inventory / monthly_sales / ratio_months

### b) SKU 健康度（独立维度 · 按进货周期分桶）

#### 进货周期 3 档（Boss 已确认）

| 桶 | 范围 | 业务场景 | 健康周转率阈值 | 业务解释 |
|---|---|---|---|---|
| **短** | ≤ 7 天 | 新品 / 紧急（紧急进货渠道）| **≥ 12 次/年** | < 1 月卖完才健康 |
| **正常** | 8-30 天 | 正常渠道（**多数 SKU**）| **≥ 6 次/年** | ≤ 2 月卖完 |
| **长** | 31-60 天 | 难进货产品 | **≥ 4 次/年** | ≤ 3 月卖完 |

> 没数据的 SKU 默认走「正常」。
> 进货周期来源：`仕入先管理リスト.xlsx` → `AB 商品进货周期` sheet。

#### 健康度 4 档判定

```
健康分数 = 交叉比率 = 粗利率 × 周转率

每个 SKU 跟它所在的进货周期桶比阈值 T：
  健康分数 ≥ T × 1.5     → 🟢 优秀
  T ≤ 健康分数 < T × 1.5 → 🟡 健康
  T/2 ≤ 健康分数 < T     → 🟠 注意
  健康分数 < T/2         → 🔴 死钱（资金占用）
```

#### 死钱量化

```python
死钱金额_jpy = 库存量 × 定義原価   # 每个 🔴 SKU 都标价
```

仪表盘按死钱金额降序，Boss 优先清最大头。

### c) 等级 × 健康度 联动矩阵

|  | 🟢 优秀 | 🟡 健康 | 🟠 注意 | 🔴 死钱 |
|---|---|---|---|---|
| **A** | 完美金牛 | 维持 | 库存薄需补 | A+死钱罕见，紧急排查（销量异常下滑？）|
| **B** | 提价候选 | 维持 | 监控 | 重点改善 |
| **C** | 利基保留 | 维持 | 季度审 | **改廃候选**（联动模块 ③）|
| **停售** | - | 清库存 | 清库存 | **优先清**（资金回笼大头）|

每个 SKU 都拿到 (等级, 健康度) 二维标签。

1. **NAS 型号**（如 DS220+ / DS920+ / DS1621+ / DS1821+）
   - 决定能否跑 Docker（DS x20+ 以上系列均可，**+** 后缀机型必有 Docker）
2. **DSM 版本**（在 NAS 控制面板 → 信息中心查）
   - DSM 6.x：Docker 套件
   - DSM 7.0+：**Container Manager**（套件中心搜，免费）
3. **NAS 内网 IP**（如 192.168.1.xxx）
   - 用于 SSH / 浏览器访问 NAS 控制台
4. **NAS 当前剩余空间**
   - Self-hosted Supabase 初始约 5GB；含数据 4 年估约 50GB；NAS 至少留 100GB
5. **是否已装 Docker / Container Manager**
   - 没装的话 Boss 在 DSM 套件中心一键装

#### 强烈推荐（外部访问用）

6. **Cloudflare 账号 + 自有域名**
   - 用 Cloudflare Tunnel 把 NAS 服务暴露到公网（不用开端口、自带 HTTPS、免费）
   - 没账号也能注册免费的 + 一个 $9/年的域名（namecheap）
   - 替代方案：Synology QuickConnect（免费但慢）+ DDNS

#### 选填（高级配置）

7. **公司公网 IP**（如果固定 IP，可不用 Cloudflare Tunnel 走端口转发）
8. **NAS 管理员账户 SSH 是否开启**
9. **现有 Supabase Free 的数据 dump**（迁移时用，`pg_dump` 一行命令）
10. **备份策略**（NAS 内置 Snapshot Replication / Hyper Backup 即可）

### 部署执行步骤（Boss 提供信息后我出脚本）

```
Step 1：NAS 安装 Container Manager（套件中心一键）
Step 2：SSH 进 NAS，git clone supabase/docker
Step 3：编辑 .env（DB password / JWT secret / Anon key 等）
Step 4：docker compose up -d
Step 5：Cloudflare Tunnel 创建 → cloudflared 在 NAS 上跑
Step 6：DNS 解析 supabase.<your-domain>.com → tunnel
Step 7：Streamlit Cloud secrets.toml 改 SUPABASE_URL
Step 8：现有数据 pg_dump → 新实例 pg_restore
Step 9：测试连接 + 切流
Step 10：Free Supabase 暂停（保留备份）
```

预计总耗时：**4-6 小时**（不含等 NAS 下载镜像）。
